PRISM: PROGRAMMABLE INTERFACE FOR STATISTICAL & SIMULATION MODELS

2 Application Programming Interface (API)

This section is geared towards people who are less familiar with APIs, or who want a refresher. If you are comfortable with APIs and don‚Äôt need a review, you can skip to the next section.

What is an API?

    Client
You are the Client

Suppose you are at home, on your laptop, and you want to use a cool new piece of software that you found on the internet. Let's say the software is written in the R language. What if you don't have R on your computer? You could install R, but maybe you don't want to. Or what if you actually want to use the software in a different language, like Python? Or maybe you want to use it in several different languages? This is what an API is for.

    Server
PRISM is on the Server

Server is a word used a lot in programming, and all it means is computer. In this context, the server is the computer you will be connecting to; it is the computer with the software that you want to use.


API communicates

We can use the API to communicate between the Client and the Server. In the example above, you could use the API to use an R software package that lives on a computer (Server) somewhere else, without having to download or use R on your computer.


REST

REST stands for REpresentational State Transfer. Ah yes, clear as mud. I understand the words individually, but‚Ä¶

REST is basically the rules for how the client (your computer) can talk to the server (the computer with PRISM, say). It is the rules for the API.

REST is a simple language. It isn‚Äôt like English, or French, or Mandarin. It only has 4 words! So much easier to be bilingual‚Ä¶

The 4 words are: GET, POST, PUT, and DELETE. Simple, right? It turns out, the difficulty comes in defining the words. The official definitions are fairly broad and rather vague, when in practice the use of the words depend on: a) context and b) general rules of thumb. A bit like English after all!

GET

When using an API, you have 2 options: you may send some data, and receive something back, or you can just receive something back. GET is the function where you just receive data. For this tutorial, we are going to use the Google Books API, which I chose because it has good documentation and doesn‚Äôt require you to get an API key.

Google Books is a huge database of books. The database lives on a computer somewhere else, called the Server. You, at home on your computer (Client), would like to see if you can find a book using Google‚Äôs software. Specifically, you want to find the book ‚ÄúIntroduction to Quantum Mechanics‚Äù by David J. Griffiths. How will you find it?

You will use a GET request. First, you need to know how to write the query. Here are a couple of rules:

  • space = ‚Äú+‚Äù
  • use ‚Äú?q=‚Äù to start the query
  • use ‚Äúintitle:‚Äù to search for a word in the title
  • use ‚Äúinauthor:‚Äù to search for a word in the author name
  • separate everything else with ‚Äú+‚Äù

Okay, let‚Äôs get started:

‚Äú?q=‚Äù to start the query
‚Äúintitle:Introduction+intitle:to+intitle:Quantum+intitle:Mechanics‚Äù to search for our title
‚Äúinauthor:David+inauthor:Griffiths‚Äù to search for our author

Putting them all together:

‚Äú?q=intitle:Introduction+intitle:to+intitle:Quantum+intitle:Mechanics+inauthor:David+inauthor:Griffiths‚Äù

Now that we have made our query, we are ready to make our GET request. Below is the code you will need to do this. In the top right corner, you can select the language which you prefer to use.

Now, for the purpose of this tutorial, we won‚Äôt use the Twitter API or the Google Books API, because they require an authentication token to complete the POST requests. Basically, if you want to POST a tweet, Twitter needs to know that it is you asking to post on your account, and not someone else. However, getting an authentication token can take a bit of time, as you have to set up a developer account etc.

In data science, sometimes a POST request looks a bit different than the Twitter POST request. For example, let‚Äôs say there is a machine learning algorithm which is able to tell you if a picture is a Cat or a Dog. This algorithm is on a Server somewhere else, and you want to use the API to check if your photo is a Cat or a Dog. This would also be an example of a POST request. You would send the data, which would be the picture, using a POST call. The API would communicate with the Server, and the algorithm would calculate if the picture was a Cat or a Dog. Then the algorithm would send back data, say a JSON file with the text ‚Äúimage: Cat‚Äù. Now you know that your image is a Cat!

PUT

The PUT request is similar to POST in that you can send and receive data. However, the PUT request is used less often. The reason for this is what computer scientists and mathematicians call ‚Äúidempotency.‚Äù An action is idempotent if repeating the action results in the same end result. Still kind of complicated, right?

Let‚Äôs do an example. The POST request is not idempotent. If you make a post on Twitter today, saying, ‚ÄúI love tacos!‚Äù, and then later on you make another post that says ‚ÄúI love tacos!‚Äù, both of those posts will stay on Twitter. The second post won‚Äôt erase the first post [even though people already know you love tacos]. You can post ‚ÄúI love tacos!‚Äù as many times as you want, and Twitter will just keep adding it to your status.

On the other hand, the PUT request is idempotent.
